## 2. 핵심 규명 (사용자 오해 정리) ★중요
- **Streamlit은 두 가지**: ① **호스팅**(share.streamlit.io, `.streamlit.app`) ② **프레임워크**(앱을 만든 재료).
  이미 ①은 안 씀(**Cloudtype 국내 호스팅으로 이전 완료**). ②는 앱 전체가 이걸로 만들어져 있어 못 뗌(= 재작성해야 사라짐, 비권장).
- **화면에 뜨는 "Streamlit 로딩"의 정체 = 서버가 잠에서 깨는 동안 뜨는 부팅 스플래시.**
  CSS로 못 없앰(앱이 아직 안 켜졌으니까). → **"로고 문제(2)"와 "느림 문제(1)"는 사실상 같은 문제.**
  **서버가 빠르게 깨어나면 그 로딩이 휙 지나가 안 보임.**
- **Streamlit 기본 UI 숨김(2번)은 이미 코드에 다 되어 있음** — `app_v2.py` 93~99줄 CSS:
  `#MainMenu`·`stToolbar`(Deploy)·`stStatusWidget`(Running…)·`stDecoration`(무지개바)·`footer`(Made with Streamlit) 전부 숨김.
  → **앱이 다 켜지면 Streamlit 흔적 거의 없음.** 추가 작업 불필요.

## 5. 속도 문제의 진짜 한계 + 권장 (사용자에게 안내함)
- 자동화로 콜드스타트를 **많이** 줄이지만 **100%는 못 없앰**:
  Cloudtype **무료 = 하루 1회 강제중지 + 몇 분 미사용 시 절전**이 근본. GitHub cron도 몇 분씩 지연될 수 있음.
- 용도별 권장:
  - **원서에 낸 링크(심사자가 아무 때나 클릭)** → 확실히 하려면 **Cloudtype 유료(월 몇천 원)**: 절전·강제중지 자체가 사라져 항상 즉시 뜸. **원서용이면 이게 최선.**
  - **발표 등 내가 보여줄 때** → 발표 **직전 링크 한 번 열어** 깨워두면 그 뒤론 빠름(무료로 충분).
- **사용자 본인 작업**은 배포서버 대신 로컬 `python -m streamlit run app_v2.py` 가 이 문제와 무관·최선.
